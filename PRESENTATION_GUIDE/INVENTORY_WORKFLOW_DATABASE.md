# Inventory Module: Workflow & Database Architecture

ระบบ **Inventory (การจัดการคลังวัสดุและอุปกรณ์)** ควบคุมสินค้าคงคลังและการถูกเบิกไปใช้ใน Job (ใบงานซ่อม) ซึ่งสามารถแยกได้ว่า วัสดุชิ้นไหนใช้แล้วหมดไป และชิ้นไหนต้องนำกลับมาคืนเพื่อใช้งานต่อ

---

## 🔄 Workflow การทำงานของ Inventory Module

1. **การลงทะเบียนของเข้าคลัง (Stock Entry)**
   * ผู้ดูแล (Admin) จะเพิ่มรายการอะไหล่หรือเครื่องมือเข้ามาในระบบ
   * สามารถระบุ **`type`** ของวัสดุได้ว่าเป็นประเภท:
      - **"ต้องคืน" (Returnable):** เช่น เครื่องเจาะ, บันได, สว่าน (คือใช้หน้างานเสร็จ ต้องนำกลับเข้าคลัง)
      - **"ไม่ต้องคืน" (Consumable):** เช่น เทปกาว, น็อต, สกรู, สายไฟ (คือเบิกไปใช้แล้วใช้เลย หมดไป)

2. **การเบิกใช้ในงานซ่อม (Checkout & Job Allocation)**
   * เมื่อมีการร่างใบงาน (Job Creation) หรือแก้ไขใบงาน จะสามารถร้องขออุปกรณ์ `usedInventory` พ่วงเข้าไปได้
   * เมื่อใบงานได้รับการอนุมัติเริ่มทำงาน สินค้าคงคลัง (`quantity`) จะถูกตัดยอดออกตามจำนวนที่เบิกใช้
   * ระบบสามารถแจ้งเตือนสถานะความพร้อมอัตโนมัติ 

3. **การคืนเข้าคลังเมื่องานเสร็จสิ้น (Restocking Flow)**
   * เมื่อช่างทำงานเสร็จและกดยืนยัน **"คำขอจบงาน"** ระบบจะได้รับการตรวจสอบโดยหัวหน้า (Lead Technician)
   * เมื่อหัวหน้ากดปุ่ม **"อนุมัติจบงาน (Approve)"** ระบบ Backend จะวิ่งไปตรวจสอบรายการของในใบงานนั้นอัตโนมัติ
   * **สำหรับ `type` "ต้องคืน"**: จำนวนของที่เบิกไป จะโดนเด้ง (Restock) กลับคืนเข้าโกดัง `Inventory` อัตโนมัติ (Automation trigger) เพื่อให้สามารถรันงานที่อื่นต่อได้

4. **การคำนวณสถานะและแจ้งเตือน (Status Monitoring)**
   * ระบบสามารถประเมิน Status อัตโนมัติตามจำนวณ (`quantity`)
   * เช่น ถ้า `quantity` = 0 สถานะเป็น "หมด" / ถ้าเหลือหลักหน่วย สถานะเป็น "ใกล้หมด" หรือต้องการเติมสินค้า

## 💾 โครงสร้าง Database (Prisma Schema: Inventory)

```prisma
model Inventory {
  id          String   @id @default(uuid()) // รหัสสิ่งของ
  name        String                        // ชื่อของวัสดุอุปกรณ์
  imageUrl    String?                       // รูปภาพสินค้า
  quantity    Int      @default(0)          // จำนวนคงเหลือในคลัง
  location    String?                       // สถานที่จัดเก็บ (ล็อค, โกดังที่เท่าไหร่)
  
  // -- การแยกหมวดหมู่ --
  status      String   @default("พร้อมใช้")  // สถานะภาพรวม: "พร้อมใช้", "ใก้ลหมด", "หมด"
  type        String   @default("ไม่ต้องคืน") // ประเภทการเบิกจ่าย: "ต้องคืน" (อุปกรณ์), "ไม่ต้องคืน" (อะไหล่สิ้นเปลือง)
  
  price       Float    @default(0)          // ราคาต้นทุนหรือราคาต่อหน่วย
  requireFrom String?                       // จุดสั่งซื้อ หรือ ร้านค้าประจำที่ต้องซื้อเติม
}
```

### 💡 อธิบาย Field สำคัญ (พูดให้เพื่อนฟังแบบเข้าใจง่าย)
* **`quantity` (จำนวนคงเหลือ):** นี่คือตัวเลขสำคัญที่สุด ทุกครั้งที่ Job เริ่ม มันจะหักออก และทุกครั้งที่ Job เสร็จ (เฉพาะอันที่ห้ามเบิกทิ้ง) มันจะบวกกลับคืนเข้ามา
* **`type` (ประเภทการใช้):** เป็นเงื่อนไข (Condition) ตัวหลักที่แยก Business Logic ว่า หลังจากช่างวิ่งไปซ่อมงานเสร็จ "มันต้องส่งคืนไหม?"
* โครงสร้างไม่ได้ทำความสัมพันธ์ตรง ๆ ฐานข้อมูล (Foreign key constraint) เข้ากับตาราง `Job` แบบแข็งทื่อ แต่ใช้วิธีฝัง (Embed) รหัสสินค้าที่ถูกเบิกในรูปแบบ JSON (`usedInventory`) ภายใน `Job` เลย เพื่อความยืดหยุ่นสูง ถอด/เพิ่มรายการได้สะดวก และง่ายต่อการปรับขยาย
