# 🛠️ เจาะลึก Tech Stack: ทำไมถึงใช้เทคโนโลยีเหล่านี้ (Pros & Cons)

เอกสารนี้เรียบเรียงมาเพื่อ **ตอบคำถามอาจารย์โดยเฉพาะ** ว่าเราใช้อุปกรณ์/ไลบรารีอะไร ทำไมถึงเลือกใช้ ข้อดีคืออะไร และมีความเข้าใจถึงข้อเสียของมันอย่างไรบ้างครับ

---

### 1. ⚛️ กรอบการทำงานหลัก: Next.js (App Router)
**ทำไมถึงเลือกใช้?** 
เพราะต้องการระบบ Full-Stack ภายในโปรเจกต์เดียว ไม่ต้องทำแยกฝั่ง Frontend (React) และ Backend (Express/Node) ให้ยุ่งยาก 
*   **✅ ข้อดี (Pros):** 
    *   **Server Components:** เรนเดอร์บนเซิร์ฟเวอร์ โหลดหน้าเว็บเร็วขึ้นเพราะส่งเป็น HTML ดิบๆ ลดภาระฝั่งผู้ใช้ (Client)
    *   **API Routes:** สร้าง Backend API ได้ในตัว (`/app/api/...`) รันทุกอย่างจบในพอร์ตเดียว
*   **❌ ข้อเสีย (Cons):** 
    *   โครงสร้าง App Router ค่อนข้างใหม่ เรียนรู้ยากกว่า React แบบดั้งเดิม และจะหาโค้ดตัวอย่างแก้ปัญหาในเน็ตยากกว่านิดหน่อย

---

### 2. 🗄️ การจัดการฐานข้อมูล: Prisma ORM + PostgreSQL (Neon)
**ทำไมถึงเลือกใช้?** 
การเขียน SQL ดิบๆ มักจะพิมพ์ผิดตอนเขียนโค้ดและเจอ Error ตอนรันเว็บ (Runtime) Prisma ช่วยผูกข้อมูล Database ให้กลายเป็น Type ของ TypeScript ทำให้โค้ดพังตั้งแต่ตอนพิมพ์แก้เลยถ้ามีฟิลด์ไหนผิด
*   **✅ ข้อดี (Pros):** 
    *   **Type-Safety 100%:** ถ้าฐานข้อมูลไม่มีคอลัมน์ชื่อนี้ โค้ดจะแจ้ง Error ทันที
    *   **Easy Migrations:** การกด Push Schema ช่วยสร้างตารางให้เลยโดยไม่ต้องเสียเวลาไปเขียน DDL (Data Definition Language)
*   **❌ ข้อเสีย (Cons):** 
    *   อาจเสียความเร็วจากการเขียน SQL ดิบไปบ้าง (Query Performance) เพราะ Prisma คอย Generate SQL อัตโนมัติซึ่งบางครั้งมันอาจจะคิวรี่อ้อมโลกกว่าเขียนดิบๆ

---

### 3. 🧠 การจัดการสถานะในแอป: Zustand
**ทำไมถึงเลือกใช้?** 
แทนที่จะไปใช้ Redux ที่เป็นมาตรฐานอุตสาหกรรมแต่มีโครงสร้าง (Boilerplate) เยอะเกินไป การใช้ Zustand ตอบโจทย์แอปขนาดกลาง-ใหญ่ได้ไวกว่า
*   **✅ ข้อดี (Pros):** 
    *   เบามาก (Lightweight) และใช้งานง่าย ไม่ต้องสร้าง Context Provider ครอบโค้ดทั้งแอปให้รก
*   **❌ ข้อเสีย (Cons):** 
    *   เครื่องมือดีบัก (Redux DevTools) อาจจะไม่ทรงพลังเท่ากับตอนที่ใช้ Redux เพียวๆ

---

### 4. 🔐 ระบบยืนยันตัวตนระดับองค์กร: NextAuth.js
**ทำไมถึงเลือกใช้?** 
ความปลอดภัยในรหัสผ่านคือหัวใจหลัก การเขียนระบบล็อกอินเองอาจจะมีช่องโหว่ (เช่น คุกกี้หายตกระหว่างทาง)
*   **✅ ข้อดี (Pros):** 
    *   รองรับ JWT (JSON Web Tokens) ออกแบบมาให้เข้ากับ Next.js แบบไร้รอยต่อ และจัดการ Session ให้เองอย่างปลอดภัย
*   **❌ ข้อเสีย (Cons):** 
    *   การปรับแต่ง (Custom) ตัวรับค่า Session ให้ครอบคลุมลึกถึงตารางของตัวเองค่อนข้างตั้งค่าซับซ้อน

---

### 5. 🤖 ปัญญาประดิษฐ์ในระบบ: Google Gemini API
**ทำไมถึงเลือกใช้?** 
เพื่อยกระดับแอปจากแค่ระบบจัดการข้อมูลธรรมดา เป็น **"ระบบจัดการข้อมูลสวมสมอง AI (Database-Aware Agent)"**
*   **✅ ข้อดี (Pros):** 
    *   รองรับ Context Window มหาศาล ทำให้เราสามารถเขียนโค้ดหลังบ้านดูดข้อมูล Jobs กับ Inventory ทั้งตารางโยนเข้าไปให้ AI ประมวลผลและตอบคำถามได้อย่างแม่นยำ
*   **❌ ข้อเสีย (Cons):** 
    *   เราต้องระวังเรื่อง Privacy (ลบข้อมูลส่วนตัวก่อนโยนให้ AI ดู) และยังมีข้อจำกัดเรื่องโควต้า RPM (Request Per Minute) บน Free Tier (จำกัดที่ 5 ครั้งต่อนาที)

---

### 6. 📄 มาตรฐานเอกสาร API: Swagger (OpenAPI)
**ทำไมถึงเลือกใช้?** 
เพื่อลดช่องว่างเวลาส่งมอบงาน หากอาจารย์หรือเพื่อนนำไปพัฒนาต่อ จะได้ไม่ต้องสุ่มเทส API เอง
*   **✅ ข้อดี (Pros):** 
    *   สามารถเจาะเข้ามาที่ `/api-doc` แล้วกด "Try it out" เพื่อคุยกับ Database ได้ทันที ทำให้ดูเป็นงานระดับ Pro-level Backend
*   **❌ ข้อเสีย (Cons):** 
    *   ถ้าโค้ดมีการเปลี่ยนแปลง ฝั่งคนเขียนโค้ดต้องคอยพิมพ์แก้ไขบล็อก `@swagger` ด้วยตัวเอง (Manual sync) ทำให้บางครั้งเอกสารอาจจะไม่อัปเดตตามโค้ด
