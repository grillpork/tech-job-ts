# 🖥️ คู่มือสคริปต์พรีเซนต์รายหน้า (Page-by-Page Demo Guide)

การพรีเซนต์รอบนี้คุณทำหน้าที่เป็น **"สถาปนิกและผู้พัฒนาระบบหลังบ้าน (Backend Developer)"** ดังนั้นเวลาเปิดหน้าจอแต่ละหน้า **ไม่ต้องเน้นว่าปุ่มสีอะไรหรือจัดเลย์เอาต์ยังไง** แต่ให้ชี้ไปที่ฟีเจอร์แล้วอธิบายว่า **"เบื้องหลังมันดึงข้อมูลหรือทำงานแบบคุยกับ Database ยังไง"** 

นี่คือสคริปต์และวิธีการพูดสำหรับแต่ละหน้าครับ:

---

## 1. หน้าเข้าสู่ระบบ (Login Page)
**👉 สิ่งที่ต้องทำบนจอ:** พิมพ์ Username/Password และกด Login
**🗣️ สคริปต์พูด:**
> "สำหรับการเข้าสู่ระบบ เราไม่ได้เก็บรหัสผ่านเป็น Text ธรรมดาครับ เบื้องหลังเราใช้ไลบรารี `bcrypt` เข้ามาช่วย Hash รหัสผ่านทุกครั้ง และเมื่อ User ล็อกอินเข้ามา ระบบจะนำเช็คข้อมูลผ่าน NextAuth ออกเป็น Session JWT (JSON Web Token) ทำให้ระบบมีความปลอดภัยสูง และสามารถตรวจสอบ Role (เช่น Admin, Manager) ต่อได้ในทุกๆ API Request โดยที่ไม่ต้องคอย Query Database ใหม่ครับ"

---

## 2. หน้า Dashboard รวม (Overview KPI)
**👉 สิ่งที่ต้องทำบนจอ:** ชี้เมาส์ไปที่กล่องการ์ดตัวเลข (KPIs) และกราฟต่างๆ
**🗣️ สคริปต์พูด:**
> "หน้านี้ข้อมูลถูกดึงมาอย่างรวดเร็วเพราะเราแยกชั้นระดับ API ครับ เบื้องหลังหน้า Dashboard นี้คือการสั่ง Aggregate Database ผ่าน Prisma โดยตรง โดยเราไม่ต้องโหลดข้อมูลดิบทั้งหมดมาหนักที่หน้าเว็บ แต่ตัว Database จะเป็นฝ่ายบวกลบเลข เช่น จำนวนงานทั้งหมด หรือ จำนวนเงิน แล้วส่งมาแค่ผลลัพธ์ที่เป็นตัวเลข ทำให้หน้า Load ได้ไวมากและไม่กิน RAM ฝั่งผู้ใช้ครับ"

---

## 3. หน้าจัดการพนักงาน (Users Management)
**👉 สิ่งที่ต้องทำบนจอ:** เปิดฟอร์มสร้างพนักงานใหม่ หรือลองเลือก Role ที่ต่างกัน
**🗣️ สคริปต์พูด:**
> "ในส่วนของ User Management ระบบหลังบ้านของเราถูกออกแบบด้วยแนวคิด RBAC (Role-Based Access Control) ครับ การสร้างยูสเซอร์หนึ่งคน ข้อมูลจะถูกจัดประเภทสิทธิ์การแอกเซสทันที และเราตั้งโครงสร้างใน Schema ให้รับฟิลด์ข้อมูลพนักงานได้ครบถ้วน เช่น Department เอย สกิลเอย และที่สำคัญการอัปโหลดรูปลงในหน้านี้ จะเป็นการบันทึกเป็น URL Link ที่ผ่านระบบอัปโหลดกลาง (File Upload Route API) แทนที่จะเก็บเป็น Local File ทำให้ฐานข้อมูลกระชับมากๆ ครับ"

---

## 4. หน้าจัดการใบงาน (Job / Task Management)
**👉 สิ่งที่ต้องทำบนจอ:** ลองโชว์การสร้างหน้าที่สั่งงาน โชว์ให้ดูว่ามีการ Assign พนักงานหลายคนได้ 
**🗣️ สคริปต์พูด:**
> "จุดที่น่าสนใจของหน้าสร้างใบงานนี้คือ ความสามารถของ Backend ที่รองรับ Relation (ความสัมพันธ์ของ Database) ครับ ในใบงาน 1 ใบ (Job) เราใช้ Prisma โยงโครงสร้างไปหาตาราง User ถึง 3 แบบใน Record เดียวกัน คือ: 1. ผู้สร้าง (Creator), 2. หัวหน้าช่าง (Lead Technician) และ 3. ลูกทีมแบบ Array (Assigned Employees) นอกจากนี้ พวก Task ย่อยๆ สเต็ปงานต่างๆ จะถูกบันทึกเป็น Table จัดลำดับความสำคัญได้แบบไม่มีปัญหาเรื่อง Data Redundancy เลยครับ"

---

## 5. หน้าคลังอุปกรณ์ และ การเบิกจ่าย (Inventory & Approval) ***(จุดไฮไลต์! โชว์ของ)***
**👉 สิ่งที่ต้องทำบนจอ:** ลองจำลองกรณี สร้างใบงานดึงเอาวัสดุมาใช้ -> สลับมาหน้า Inventory ไปที่แท็บ "คำขอเบิกวัสดุ" -> กด **"อนุมัติ"**
**🗣️ สคริปต์พูด (เน้นเสียงมั่นใจ):**
> "มาถึงจุดที่เป็น Core Logic ของเราครับ นั่นคือระบบการเบิกจ่ายอะไหล่ ตอนที่ช่างขอเบิกของ ระบบสร้าง Request แจ้ง Admin ทันที (Real-time tracking) พอ Admin กด 'อนุมัติ' ปุ๊บ โค้ดหลังบ้านเราจะเริ่มทำงานครับ...
> 
> เริ่มจากการ **Validate สต๊อกก่อน** ถ้าอุปกรณ์มีไม่พอ มันจะ Block เด้งเตือนทันทีไม่ให้เลขติดลบ! แต่ถ้ามีพอ ระบบจะทำ Database Transaction วิ่งไปหักสต๊อกพร้อมกับเปลี่ยน Status ของไอเท็มชิ้นนั้น (เช่นจาก 'พร้อมใช้' เป็น 'ใกล้หมด') ภายในเสี้ยววินาที ข้อมูลนี้ลง PostgreSQL ถาวร รีเฟรชกี่รอบก็ของไม่หายหรือเพี้ยนครับ!"

---

## 6. หน้าพิจารณาปิดงาน และ คืนของ (Job Completion & Auto-Return) ***(จุดโชว์ความสมาร์ท)***
**👉 สิ่งที่ต้องทำบนจอ:** ไปหน้า Request ขอจบงาน แล้วกด **"อนุมัติจบงาน"**
**🗣️ สคริปต์พูด:**
> "อีกหนึ่งระบบ Automation หลังบ้าน ที่ออกแบบเพื่อช่วยลดภาระพนักงานบัญชี/แอดมินครับ เมื่อช่างทำงานเสร็จแล้วทำการปิดจ๊อบ สิ่งที่ Backend API เราทำเบื้องหลังคือ จะเข้ากระบวนการกรองข้อมูล (Filter) ขยายดูว่าในใบงานนี้ ช่างเบิกอะไรไปบ้าง จากนั้นเช็ค Database หมวด Equipment Rule ครับ 
> 
> ถ้าประเภทตั้งค่าไว้ว่า **'ต้องคืน'** (เช่น สว่าน อะไหล่ยืม) Database จะรันคำสั่ง Increment จำนวนสต๊อกบวกกลับเข้าคลังให้อัตโนมัติเป๊ะๆ! ส่วนของประเภท 'ไม่ต้องคืน' (เช่น น็อต พันสายไฟ) จะถูกข้ามไปเลย ทำให้การตัด-คืนสต๊อกมีความสมบูรณ์แบบ 100% ครับ!"

---

## 7. หน้าบันทึกระบบ (Audit Logs / Reports)
**👉 สิ่งที่ต้องทำบนจอ:** เลื่อนเพื่อดูประวัติการเปลี่ยนแปลงต่างๆ (ถ้ามี)
**🗣️ สคริปต์พูด:**
> "สุดท้ายเพื่อความโปร่งใสและตรวจสอบย้อนหลัง เรามี Database Model ที่แยกตัวออกมาระบุข้อมูล Audit Logger ครับ ทุกๆ การสร้าง ทุบ ลบ อัพเดต ไม่ว่าพนักงานจะกด Approve พลาด หรือลบอะไรไปก็ตาม ตัว Backend API จะ Inject (แทรก) ประวัติฝังลงไปใน Database เสมอ เพื่อแสดงรายละเอียดแบบเรียลไทม์ว่า ใครทำอะไร กับ entity ไหน สังเกตเอาไปใช้เป็นรายงานสำหรับ Manager ต่อไปได้สบายๆ ครับ"

---

## 🎯 ทริคในการเอาตัวรอดหน้างาน 🎯
1. **ถ้าถูกถามเรื่อง "อากระตุก หรือ โหลดช้า":** ให้ตอบว่าเรา Deploy ของไปที่ Server ฟรีชั่วคราว มี Cold Boot สตาร์ทบ้าง แต่โครงสร้างฐานข้อมูล (Postgres) รองรับ Concurrency (คนใช้งานพร้อมกัน) สบายมาก
2. **หากกดแล้วมี Error แดงขึ้นแบบไม่คาดฝัน:** ให้มีสติ อย่าลนลาน! ให้ตอบนิ่งๆ ว่า *"อ๋อ อันนี้เป็นฝั่ง Frontend แสดง Error ไว้จากการทำ Input Validation ดักไว้ครับ ทำให้ Request ไม่วิ่งไปชน Backend ป้องกัน Data ขยะหลุดเข้าไปในระบบครับ เราวางระบบ Validate สองชั้นไว้"*
3. **อย่าอยู่หน้าเดียวนาน:** ให้เตรียมข้อมูลจำลอง (Mock) ไว้ล่วงหน้า จะได้มากดคลิ๊กโชว์ไหลลื่นๆ ไม่ต้องมานั่งเสิร์ชหาข้อมูล หรือพิมพ์ชื่อตอนพรีเซนต์
